# 🚀 Project: Kronos

> **Kronos is an extensible algorithmic trading platform capable of running multiple strategies concurrently across one or more brokers with real-time market data and configurable risk management.**

---

# Development Rules

Before we write any code, let's define a few rules that we won't break.

## Rule 1: Strategy never talks to Broker

❌ Wrong

```ts
strategy.buy();
```

✅ Correct

```text
Strategy
    ↓
Signal
    ↓
Risk Engine
    ↓
Order Engine
    ↓
Broker
```

This allows us to add logging, risk checks, paper trading, etc., without changing strategies.

---

## Rule 2: Broker is Replaceable

The Strategy should never know whether it's using:

* Kotak
* Zerodha
* Upstox
* Paper Broker

It only knows:

```ts
broker.placeOrder(...)
```

---

## Rule 3: Everything has one responsibility

Instead of:

```text
MarketService

↓

Gets Tick

Calculates EMA

Places Order

Stores DB
```

We'll do:

```text
Market Engine

↓

Strategy Engine

↓

Risk Engine

↓

Order Engine
```

Each module has one clear job.

---

## Rule 4: Strategy is a Plugin

Tomorrow we should be able to add:

```
EMA Strategy
```

without changing the engine.

Next week:

```
ORB Strategy
```

Same engine.

# One Improvement Before We Start

There is **one thing** I want to add because I think it will save us a lot of headaches later.

## `packages/core`

Instead of putting the Event Bus, lifecycle management, and dependency injection logic into random places, I'd like a dedicated `core` package.

It would contain things like:

```
packages/core

Event Bus

Lifecycle Manager

Dependency Container

Base Services

Application Context
```

This isn't about making things fancy. It's about having a single place for the infrastructure that both the API and Worker share, instead of duplicating it later.

# 🎯 Our First Coding Task

I don't want our first commit to be "create-next-app."

Instead, our first milestone will be:

**"Kronos boots successfully."**

That means:

* Turborepo is initialized.
* Frontend, API, and Worker all start with one command.
* PostgreSQL and Redis start with Docker Compose.
* Shared packages compile correctly.
* Path aliases work.
* Hot reload works across the monorepo.
* A simple `/health` endpoint in the API responds.
* The Worker logs: `Kronos Worker Started`.

Once that foundation is in place, every feature we build afterwards will fit into a clean, well-tested architecture instead of being bolted on. I think that's the best way to start a project we expect to grow over time.
